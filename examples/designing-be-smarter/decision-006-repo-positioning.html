Where does /be-smarter live, and what is it to decision-kit?

You asked for a new repo with decision-kit's license and contribution method. The real decision underneath: what's the relationship between the two projects — and does the name hold up?

Builds on Decisions 1–5: the skill is standard-conformant and headless-first — a different premise from decision-kit's judgment-first gate. The repo structure should make that difference legible, not blur it.
Research Context
  • The name is free. No meaningful "be-smarter" collisions on GitHub (top search hits are unrelated). The skill-command form /be-smarter is imperative, memorable, and states the promise — consistent with decision-kit's verb-style commands (/decide, /strategize, /excavate).
  • Decision-kit's scaffolding is designed to be portable. Apache 2.0 + CODE_OF_CONDUCT + CONTRIBUTING (issue-first, one-focused-PR, dogfood-your-.decisions) + SPEC.md with explicit rules + examples/ — nothing in it is decision-kit-specific. Porting is a copy-and-edit, not a design effort. decision-kit repo
  • The philosophical difference is real and worth preserving. Decision-kit's SPEC forbids skills that "bypass the decision gate" — human judgment is the point. /be-smarter's default is headless. Housing a gate-bypassing skill inside the repo whose spec forbids gate-bypassing creates a contradiction contributors will trip on.
  • The Agent Skills spec rewards standalone skill repos. Install is "copy this directory into your host's skills folder" — a repo whose root IS the skill directory makes that one command. Monorepos add a path layer to every install instruction. Agent Skills spec, Codex skills docs
  • Cross-pollination needs only two links: decision-kit's /decide orchestrator can route "make this smarter"-shaped intents to /be-smarter when installed; /be-smarter's review mode emits decisions.json-compatible traces that decision-kit's action skills consume (Decision 4). Integration is protocol-level, not repo-level.
Options
A

Inside decision-kit

A new top-level meta/be-smarter/ category in the existing repo. One community, shared scaffolding, zero setup.

decision-kit/ ├── thinking/ · action/ · orchestrator/ └── meta/ └── be-smarter/ ← new category

Zero scaffolding work; one community and issue tracker

/decide routing is in-repo and trivial

Contradicts the SPEC's no-gate-bypass rule in its own house

Buries the 32-tool standalone story under decision-kit's Claude-centric framing — and ignores your explicit ask

★ Recommended
B

Sibling repo: jnemargut/be-smarter

A standalone repo whose root is the installable skill, with decision-kit's governance ported verbatim (Apache 2.0, CONTRIBUTING, CODE_OF_CONDUCT, SPEC pattern, examples/). READMEs cross-link as a family: "decision-kit puts you in the loop; be-smarter thinks that way even when you're not."

be-smarter/ ← repo root = skill dir ├── SKILL.md (host-neutral, spec-conformant) ├── PROTOCOL.md (paste-anywhere version) ├── references/ (triage rubric, trace schema, dial) ├── examples/ (incl. this .decisions/ run) ├── SPEC.md · CONTRIBUTING.md · LICENSE (Apache 2.0) └── README.md (install = one cp per host)

Matches your ask exactly; one-command install per host

Each repo keeps a coherent philosophy — no spec contradiction

Standalone story for the 32-tool audience who've never heard of decision-kit

Second issue tracker, second README to keep current

Governance files duplicated (drift risk — mitigate by porting verbatim and rarely touching)

C

Umbrella org for a kit family

Create a GitHub org (e.g. decisionkit-ai), move decision-kit in, add be-smarter beside it, share org-level governance and a family brand.

decisionkit-ai/ ← org ├── decision-kit ├── be-smarter └── .github/ (shared governance)

Cleanest long-term home if more kits are coming

Org-level community files end the duplication problem

Moving decision-kit breaks existing links/installs for a 1-star repo's benefit

Brand infrastructure before product traction — ceremony first, users later

D

Fork decision-kit and trim

Literal interpretation of "same contribution method": fork the repo, delete everything but scaffolding, build be-smarter in the shell. History and governance carry over automatically.

be-smarter/ (fork of decision-kit) ├── [decision-kit history preserved] └── everything deleted except governance

Governance fidelity is automatic, not manual

GitHub marks it "forked from decision-kit" forever — wrong signal for a standalone product

Irrelevant history pollutes blame/log; forks get worse discovery treatment
